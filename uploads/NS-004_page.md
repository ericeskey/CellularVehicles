# The Output

## Episode Summary

A sequencing run can finish successfully and still leave the work unfinished.

In this episode, the machine has done its job. The data exists. Files have been produced. Yet the next step does not move forward as cleanly as expected, because the workflow behind the library does not match the assumptions built into the output.

The core question is where a sequencing workflow actually ends. Smart teams may assume the finish line is the generated file. This episode follows the moment when that assumption starts to wobble.

The discovery is restrained but important: users may not want the machine to perform all downstream analysis. They may want something more practical—a way for the machine to recognize common protocol differences and produce output that fits the work already being done.

## In This Episode

- Why “the run is complete” does not always mean the result is usable.
    
- How changing protocols expose hidden assumptions in machine output.
    
- Why users may distrust broad built-in analysis while still wanting smarter output handling.
    
- Where rigid demultiplexing pushes work outside the machine.
    
- How a finished file can still leave the workflow unresolved.
    

## Why This Matters

Software roadmaps often drift toward bigger promises: more analysis, more automation, more end-to-end intelligence. This episode points in a more disciplined direction.

For users like Martin, the opportunity is not to make the machine pretend every lab works the same way. It is to help the output respect the differences that already exist.

## Where This Fits In The Series

Episode 1 began before the run, where a setup file could stop progress. Episode 2 moved into the timing pressure before sequencing. Episode 3 paused after the run to show how trusted machines earn forgiveness for burdensome maintenance. Episode 4 closes the series at the boundary between generated data and usable output, where the workflow continues beyond the machine.
