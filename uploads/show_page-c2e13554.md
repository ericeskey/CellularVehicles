# When the Machine Works, but the Workflow Breaks

## About This Series

What happens when a sequencing workflow succeeds only because the right person is there to absorb everything the system cannot handle?

This series follows Martin, a sequencing operator working inside a translational oncology lab, where patient material, prepared libraries, machine setup, data output, and analysis all have to stay connected. The machine does many things well. It guides users through steps. It produces trusted data. It gives the lab a way to test before committing to larger, more expensive runs.

At first, the problem looks like ordinary friction: a sample-sheet error, a fragile chemistry step, a cumbersome wash, an output file that needs extra handling. Each issue appears small enough to tolerate. Then the pattern comes into focus. The workflow depends on invisible rescue work: expertise, timing, memory, manual correction, and trust.

The series moves through four incidents. A file stops the run before it begins. A cooled library waits against the clock. A washing cartridge reveals why users forgive some burdens. An output file shows where the machine’s assumptions end.

By the end, listeners will hear sequencing differently. Throughput is not only run time. Usability is not only guidance. Reliability is not only data quality. The work holds together when the machine, the chemistry, the people, and the data remain in continuity.

## Episode Guide

### Episode 1 — The File

A sample sheet should be the last administrative step before a run. Instead, it becomes the place where the workflow stops and Martin’s hidden expertise enters.

### Episode 2 — Fresh

A denatured library sits on ice while the clock starts to matter. The episode follows the fragile bridge between prepared material and a machine ready to run.

### Episode 3 — The Wash

The washing cartridge is repetitive, manual, and easy to resent. Yet it also reveals why a trusted machine can earn forgiveness for work users would rather not do.

### Episode 4 — The Output

The run finishes, but the result is not always finished. A file at the end of the process shows how changing protocols can push the workflow beyond the machine.

## What You'll Learn

- Why small setup failures can create large workflow dependencies.
    
- How “expert rescue” hides inside otherwise functional systems.
    
- Why biological timing and software friction cannot be treated separately.
    
- How trusted data changes what users tolerate.
    
- Why faster turnaround matters because it preserves focus, not just time.
    
- Where users want smarter software—and where they do not.
    
- How a machine can work exactly as designed while the workflow still breaks.
    

## Recommended Starting Point

Start with Episode 1. The sample-sheet incident is the doorway into the whole series: a small file, a stalled run, and the first sign that the workflow depends on more than the machine.
