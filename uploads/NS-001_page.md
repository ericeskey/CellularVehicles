# The File

## Episode Summary

A sequencing run can stop before the machine ever begins.

In this episode, a sample sheet appears to be a routine setup file. The libraries are ready, the machine is waiting, and the next step should be simple. Then the screen refuses to continue.

The failure is small, familiar, and strangely powerful. The machine can detect that something is wrong, but it cannot help the user understand what to fix. Progress resumes only when Martin steps in and recognizes one of the common mistakes hidden inside the file.

The episode follows that interruption until it reveals a larger dependency: the workflow works because someone nearby knows how to absorb the friction.

## In This Episode

- Why ordinary setup errors can stop sophisticated scientific work.
    
- How cryptic machine feedback turns novice users into dependent users.
    
- Why “easy to fix” does not mean “easy to recover from.”
    
- How expert rescue hides inside otherwise functional workflows.
    
- Where a sample sheet becomes more than an administrative file.
    

## Why This Matters

The episode challenges a common assumption about usability: that a system is easy to use if it guides people when everything goes right. The harder test comes when something ordinary goes wrong.

When recovery depends on Martin, the system has not fully delegated the work. It has delegated the steps, while keeping the interpretation of failure in someone’s head.

## Where This Fits In The Series

This is the opening episode of _When the Machine Works, but the Workflow Breaks_. It begins with the smallest visible object in the workflow: a file. The next episode moves from software friction to biological timing, where delay does not just interrupt progress; it can change what must be done next.
